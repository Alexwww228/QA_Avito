# Тест-кейсы для API микросервиса объявлений (Задание 2.1)

**Host:** `https://qa-internship.avito.com`  
**Версия API:** `/api/1` (основная по условию задания)  
**Эндпоинты (4 ручки):**
- `POST /api/1/item` — создать объявление
- `GET /api/1/item/{id}` — получить объявление по ID (UUID)
- `GET /api/1/{sellerID}/item` — получить все объявления продавца
- `GET /api/1/statistic/{id}` — получить статистику по ID объявления

**Особенности API (выявлено при исследовании):**
- ID объявления — UUID (строка).
- При GET `/item/{id}` и `/statistic/{id}` возвращается **массив** с 1 элементом.
- При GET `/sellerID/item` — массив (может быть пустым или содержать элементы других тестеров).
- Ошибки возвращаются в формате `{"result": {"message": "...", "messages": {...}}, "status": "400/404"}`.
- `sellerID` — целое число 111111–999999 (используется random для избежания коллизий).

**Техники тест-дизайна:** классы эквивалентности, граничные значения, error guessing, pairwise, state transition (E2E), idempotency check.

## 1. Позитивные сценарии
**TC-01** Создать объявление с валидными данными (минимальные значения)  
**TC-02** Создать объявление с максимальными значениями (price=2147483647, большие статистики)  
**TC-03** Создать несколько объявлений с одинаковыми данными → разные ID (неидемпотентность)  

**TC-04** E2E: Создать → получить по ID (совпадение всех полей)  
**TC-05** E2E: Создать → получить статистику (совпадение statistics)  
**TC-06** E2E: Создать 2 объявления с одним sellerID → получить по sellerID (в списке оба наших ID)  

## 2. Негативные сценарии
**TC-07** Создать без поля `sellerID` → 400  
**TC-08** Создать с `sellerID` строкой / отрицательным / вне диапазона → 400  
**TC-09** Создать с отрицательным `price` / пустым `name` → 400  
**TC-10** Создать с некорректным JSON / лишними полями → 400  

**TC-11** GET `/item/{invalid-uuid}` → 400 "ID айтема не UUID"  
**TC-12** GET `/item/{valid-uuid-but-not-exist}` → 404 "item ... not found"  
**TC-13** GET `/statistic/{invalid-uuid}` → 400  
**TC-14** GET `/statistic/{not-exist-uuid}` → 404  

**TC-15** GET `/{sellerID}/item` с нечисловым sellerID → 400  

## 3. Корнер-кейсы
**TC-16** price = 0  
**TC-17** statistics со всеми нулями / максимальными значениями  
**TC-18** name = "" (пустая строка)  
**TC-19** Очень длинное name (1000+ символов)  
**TC-20** Idempotency: повторный CREATE с теми же данными → новый ID  

## 4. Нефункциональные проверки
- Время ответа < 2000 мс для всех эндпоинтов
- Корректный Content-Type: application/json
- Корректная структура ошибок
- Нет утечек данных других продавцов в GET по ID

Все кейсы автоматизированы в `test_api.py` (E2E + Allure-шаги).
